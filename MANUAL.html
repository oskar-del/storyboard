Storyboard — Feature Manual

Everything in the dashboard, why it exists, and how it connects. Built during a 10-day build-in-public sprint with Claude in Cowork mode.
📅 Sprint: Apr 15–24 2026 🗂 Day 6 of 10 🌐 oskar-del.github.io/storyboard
Foundation

The core idea

Every time you start a new AI session, you re-explain the context, re-make the same decisions, re-discover the dead ends. Storyboard fixes this by capturing everything — sessions, decisions, ideas, rejections — and handing it all to the next session in one click.
The key insight: AI tools reset on every session. Your project doesn't. Storyboard is the persistent memory layer that sits above all your AI tools — Claude, ChatGPT, Gemini, Cursor — and holds intent across every context window, every tool, every day.
Built with Storyboard: This product was built using itself. Every session was captured, every decision logged, every rejection stored. The sprint accelerated because the context got richer every day — that's the compound effect the product promises.
Core concept

Block types — the vocabulary

Every piece of work becomes a block. Blocks have types. Type defines the color, icon, and meaning — not the project. This means you can scan a feed and instantly know what happened without reading every title.
Type Color What it represents When to use
Session Purple A complete AI work session — compacted into decisions + ideas + summary Auto-created by MCP server or Chrome extension at end of session
Idea Amber A captured idea — not yet decided, not rejected Quick capture, inbox review, ⌘K search
Decision Green A definitive choice made — architecture, product, strategy Extracted from sessions or logged manually
Discussion Blue A thinking document — exploration of a topic without a conclusion yet Named discussions saved to /discussions/ folder
Rejection Red A path explicitly ruled out — with the reason captured Prevents re-exploring dead ends. "We tried this. Here's why it failed."
Skill Lavender A reusable prompt pattern — a workflow that can be triggered again From Skills view; skills are scored and tracked over time
Roadmap Teal A decided feature — status: next / soon / later Ideas that got promoted to "we're building this"
Intent Rose The north star goal for a project at a point in time Set via the intent bar in project header — used to seed new sessions
Compaction Slate A mid-session context compression — what was preserved when window filled Auto-created when context gets close to limit — captures what matters
Automation Emerald A live automation — something running without manual input Visible in Pipeline view Stage 4 and project drawer
Simulation Indigo A synthetic test run — agent preview output, persona testing Agent Preview tab generates simulations
Audit Orange An AEO/SEO content audit result — scored findings Run from the Website Audit tool in Intelligence
Why type beats project for color: Before Day 6, all blocks in a project looked the same (same project color). You couldn't tell a session from a decision at a glance. Now type is primary — the stripe and icon band use the semantic color. Project is secondary — just the dot. One glance at the feed tells you what happened, not just which project it's in.

Feed controls

The period tabs (All time / Month / Week / Today) and view toggles combine to create radically different pictures of your work. Same blocks, different lens.
⊞
Grid
Default. Masonry columns. Blocks newest first. Featured block pinned at top. Most natural for browsing.
◫
Canvas
Project overview cards. Each project gets a summary card with health score, latest block, live count. Click a card → project detail drawer.
≡
Timeline
Blocks grouped by week. Shows momentum — how dense was last week vs this week. Activity histogram per day.
→
Pipeline
Idea → Roadmap → Building → Automated → Learning. See where your work sits in the build cycle. Based on block types.
📓
Journal
Narrative log — sessions ordered as readable story entries. Great for explaining the sprint to someone new.
☰
List
Compact rows. Good for scanning lots of blocks quickly or copy-pasting titles for a report.

If nothing is captured yet, you get a morning kickoff card with a "Good morning" greeting and inline textarea to log the first decision of the day.
Top controls

Topbar — the command strip

Every persistent action lives in the topbar. It stays visible regardless of which view or project you're in.
🚀
New session
Why: One-click context seeding
Opens the session bootstrap — a pre-built prompt containing your project's recent decisions, active intent, and carry-forward items. Paste into any AI tool to start with full context instead of zero.
How to use: Click → copy the generated prompt → paste into Claude/ChatGPT/Gemini
⚡
Context fill meter
Why: Prevents context loss
Estimates how full the current Claude session is based on time elapsed and blocks captured. Goes amber at ~60%, red at ~85%. Red = capture your work now before the window closes.
Click it to open Context Windows view — all sessions as numbered chapters
🔍
Search ⌘K
Why: Find anything instantly
Command-palette style search across all blocks. Type filters (session / idea / decision / discussion / rejection). Arrow key navigation. Term highlighting. Matches inside decision text, not just titles.
Keyboard: ⌘K to open, ↑↓ to navigate, Enter to open block, Esc to close
⚡
Carry forward
Why: Continue work across context windows
The most-used action. Takes the current session's decisions + intent and prepares them for the next session. Available on any block via the detail overlay. Also appears as "⚡ Carry forward" in Today digest session cards.
How to use: Open any block detail → click Carry forward → paste into new session
🌙☀️
Dark / Light mode
Why: Dark is default, light is opt-in
Dark mode is the default. Light mode was added as an explicit opt-in because most serious AI work happens at night or in terminals. Also accessible from Settings → Appearance.
Toggle: Nav mode buttons (AI = dark, OS = light) or Settings → Appearance
⚙
Settings
Why: 4 tabs of real configuration
Appearance (theme + compact mode), Data (live stats + export/import JSON + clear local), AI (Anthropic API key for inbox extraction + MCP health check), About (sprint day + GitHub link).
Compact mode: Reduces card padding globally — good on smaller screens
Context layer

Overview strip — contextual stats bar

The horizontal strip above the feed isn't static. It changes based on where you are in the app. Each section gets its own first card with relevant stats.
📊
All work view
Shows total blocks, decisions, ideas, sprint day pill, and "last captured" timestamp. The velocity card shows % change this week vs last.
📁
Project view
First card scoped to that project: block count, decisions, roadmap items, last session. Helps you orient before scrolling into the blocks.
💡
Ideas section
Shows idea count, how many projects have ideas, and which project has the most. Helps you see the idea backlog at a glance.
✓
Decisions section
Decision count, which project has the most decisions logged, and recency. Decisions are the most valuable asset — they prevent re-doing work.
Project layer

Project header — the workstream map

When you click a project in the nav, the feed gets a header card. This isn't just a label — it's a full project dashboard showing intent, stats, latest block, and crucially, the task windows strip.
🗂
Task windows strip (new — Day 6)
Why: Projects have multiple parallel workstreams that were invisible
A project like Storyboard has 6 things running at once: dashboard, landing page, MCP server, Chrome extension, mobile PWA, context visual. Previously all sessions piled into one undifferentiated project feed. Now each task is a card in the strip. Block count, session count, and last-active date visible at a glance. Green pulsing dot = automation running on that task.
Click a task card → feed filters to just that workstream → breadcrumb shows Project / task-name → click ✕ to clear. Tag blocks with task: "taskname" in blocks-data.json to populate.
🎯
Intent bar
Why: North star should be visible during every session
The project's current goal — what you're trying to achieve right now, not what the project is long-term. Click the pencil to update it. It gets included in the session bootstrap prompt automatically.
▶
Seeds (context starters)
Why: Saves writing the same context prompt repeatedly
Pre-built context prompts for common session types. Click to copy, paste into Claude. Each seed includes the project's recent decisions, intent, and relevant background.
Block layer

Block detail overlay

Click any block to open the detail overlay. It's the primary interaction for using captured context.
📝
Section order
Summary first, then decisions list, then ideas list, then related blocks. Summary is the executive version. Decisions are the actionable outputs. Related blocks help you see the thread.
🔗
Related blocks panel
The 4 most recent blocks from the same project (excluding sessions). Shows context thread — what else happened around this block. Click any related block to navigate there.
⚡
Carry forward (on ALL blocks)
Previously only available on seed=true blocks. Now every block has carry forward. This means you can resume from any decision, idea, or session — not just the ones explicitly marked for seeding.
🏷
Type chips + meta row
Below the title: type badge (colored), project badge, date, and any chips from the block. The context window fill bar shows for session blocks — lets you see how compressed the session was when captured.
Daily view

Today digest — the daily story

Clicking "Today" in the period tabs doesn't show a filtered grid — it shows a completely different layout built around the day's narrative.
🌅
Morning kickoff card
Why: The first block of the day sets the narrative
When Today has no blocks yet, instead of an empty state you get a "Good morning" card with a textarea. Type your first decision or idea, select a project, press Enter. It drops in as the first block of the day with a Tetris-style animation.
📊
Day header + stats
Date, session count, decision count, idea count. Project pills showing which projects were active today with block counts. At a glance: did this day produce anything meaningful?
✓
Flat decisions list
Every decision extracted from today's blocks, shown as a flat list — not buried inside session cards. Click any decision to navigate to its parent block. This is the daily output of the work.
🎬
Drop-in animation
Why: Makes the day feel alive, not static
When Today loads, cards drop in with a staggered animation (40ms between each). When you log a new block via the kickoff card, it animates in from above. The Tetris metaphor — blocks accumulating to build the day's story.
Intelligence layer

Intelligence tools

Beyond the feed, these tools analyze and generate from the captured data.
🤖
Agents — Agent Preview
Why: Test how an AI agent would interpret your project
Synthetic user testing. Pick a persona (power user, first-timer, mobile user, etc.) and see how they'd experience the product. Generates simulation blocks — visible in the Pipeline view Automated stage. Shows gaps you'd miss from the inside.
⚡
Context — context windows view
Why: Makes invisible context boundaries visible
Shows all sessions as numbered chapters. Each session block renders as a "context fault line" — a full-width divider showing exactly where one context window ended and the next began. Most AI users don't know where their context broke.
🌐
Website Audit
Why: AEO/SEO quality gate before publish
46-check audit across 7 categories — AEO foundations, crawlability, technical SEO. Scores 0–100. For Curated.Estate: articles score 100/100, other pages target 90+. Results save as audit blocks.
🏷
Categories
Why: Cross-project type grouping
Groups blocks across all projects by category — SEO, CRM, Design, Automation, etc. Same blocks, cross-project lens. Connected to the task windows concept: tasks are project-specific instances of cross-project categories.
Under the hood

Data model — how it all fits

Storyboard is a single HTML file + a JSON data file + an optional MCP server. Understanding the data model explains every design decision.
blocks-data.json — the source of truth. Every block is an object: { id, type, project, task, title, summary, decisions[], ideas[], chips[], ts, _source }. The task field (added Day 6) connects blocks to parallel workstreams within a project. The _source field tracks whether it came from claude.ai, the Chrome extension, mobile memo, or was typed manually.
📦
Block fields
id — unique, used for navigation and dedup
type — determines color, icon, render template
project — which project it belongs to
task — which workstream within the project
ts — timestamp as YYYYMMDD integer for fast sorting
decisions[] — extracted decisions as string array
ideas[] — extracted ideas as string array
chips[] — tags/keywords for search and filtering
_source — how it was captured
🔄
Data sources
blocks-data.json — base data, checked into git
localStorage — local blocks added via UI or quick capture
MCP server (port 3456) — live capture from Claude sessions
Chrome extension — auto-reads Claude.ai chat, posts to /raw-capture
Mobile memo — voice-to-block via mobile interface
Merged — all sources merged at render time, deduped by ID
🏗
Project metadata
PROJ{} — icon, color, background per project
PROJECT_META{} — goal, northStar, deploy status, stats, seeds
AUTOMATIONS[] — live automations with schedule and project
SKILLS[] — prompt patterns with scores and categories
These are defined as JS constants at the top of app.html.
💾
Settings / Data management
Settings → Data tab shows live storage stats. Export JSON downloads everything (base + local blocks + settings). Import JSON merges by ID — safe to run multiple times, no duplicates. Clear local removes only localStorage blocks, base data untouched.
Why it works this way

Key design decisions

Every non-obvious choice has a reason. These are the decisions that shaped the product during the sprint.
  • Dark mode is default — AI work happens at night, in terminals, on screens. Light is opt-in for presentations.
  • Block type beats project for visual color — you need to know what happened before you know where it happened.
  • Single HTML file — no build step, no npm install, works offline, can be opened from Finder. Speed to first value is instant.
  • Carry forward on ALL blocks — not just seed=true. Any block can restart a session. Constraining it to "seed blocks" was artificial.
  • Decisions extracted as flat arrays — not embedded in session prose. This makes the 241 decisions searchable, exportable, and seedable individually.
  • Task filter closes the project drawer — you don't want two levels of UI open. Close and filter is cleaner than panel + filter overlay.
  • Today digest is a completely different layout — not a filtered grid. A day has narrative structure (morning → decisions → ideas → other). The grid doesn't respect that.
  • Morning kickoff card instead of empty state — an empty state is passive. A kickoff card is a prompt. The first decision of the day deserves a better UX than "nothing here yet."
  • MCP-first architecture — browser extension covers tools without MCP (ChatGPT, Gemini). Everything routes through mcp-server.js as the universal intake for Claude.
  • GitHub Pages = demo/meetup URL — always current after every push. No separate deployment step needed.
  • Rejections are first-class blocks — not deleted ideas. Red, visible, with reasons captured. "We tried this, here's why it failed" is more valuable than silence.
  • Context fill meter shows compaction risk — users don't know when their context is close to the limit. Making it visible changes behavior: capture early, don't lose work.
  • Sprint day auto-calculated — not hardcoded. `Math.floor((now - Apr 15) / 86400000) + 1`. Day 1 = Apr 15. Caps at 10.
Day 7+

What's live — Day 6 evening additions

These features shipped during the Day 6 autonomous session and are live now.
⚡
Live block logging (Tetris mode)
Why: Blocks were only logged at end-of-session. The day's story was assembled backwards, after the fact. Tetris mode makes it real time.
How it works: MCP tool log_block + HTTP endpoint /log-block. Claude calls it the moment a decision is made. Blocks appear in the Today feed within 10 seconds via the existing 10s poll loop. SKILL.md updated with the logging protocol — Claude logs silently, one line confirm.
📖
This manual
Why: Features were accumulating faster than anyone could remember them. A friend meeting was coming up. The rule is: when a feature ships, the manual updates.
Available at MANUAL.html alongside the app. Linked from Settings → About. Linked from nav footer user menu → Feature manual. Updated whenever features ship.
